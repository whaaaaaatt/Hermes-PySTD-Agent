## 第二轮收尾：修复 `--insecure` 不生效的 bug

**用户报告**：带不带 `--insecure` 前端都报 "Failed to start: unauthorized"。

**根因**：`web/server.py` 的 `ServerState.__init__` 没读 `allow_public` 标志 —— bind `0.0.0.0` 时仍按"非 loopback 就要 token"路径走，生成 token、写入 config，**完全无视 `--insecure`**。前端拿到 `auth_required: true` + 一个用户从来没见过的 token → 401 → "Failed to start"。

**修复**：
1. `Auth.__init__(token, disabled=False)` 新增 `disabled` 字段；`required()` 第一行就 `if self.disabled: return False`
2. `ServerState.__init__` 读 `allow_public` → 若 True，`Auth(token="", disabled=True)`
3. `_check_auth` 用 `s.bind_host`（CLI 传入的真实 bind 主机），不是 `handler.headers["Host"]`（不可信）
4. `/api/status` 返回 `auth_disabled` + `insecure_mode` 字段；loopback 和 insecure 都不返回 token
5. `state.host` → `state.bind_host`（之前误用 handler 的某个 attrs）
6. 启动横幅：insecure 显示 `--insecure: NO AUTH` + 警告

**验证**：
- 单元测试 62/62 通过（新增 2 个：`test_auth_insecure_disables_token_on_lan`、`test_serverstate_insecure_skips_token_generation`）
- 端到端 socket 测试：GET/POST /api/status 都 200，**带 stale token 也 200**，host 正确为 0.0.0.0，auth_required=false

**附带的 bug**：`ServerState` 把 `self.state` 覆盖成 `StateStore` 实例，与 `handler.state`（应为 ServerState）重名，但不影响功能（handler.state 通过 class 属性显式设置）。

**没改成的事**：端口残留进程（之前的测试 server 没正常退出），需要用 taskkill 清理；建议 `web` 子命令的 ThreadedHTTPServer 在启动失败时优雅关闭、文档化 `Ctrl-C` 后的退出语义。

## 第四轮：前端重构 + 修复聊天 NPE

**用户报两个问题**：
1. 前端布局不合理——所有 tab 内容塞在左侧 280px 边栏里太挤；tools/skills 不可点击查看详情
2. 聊天时报错：`Uncaught (in promise) TypeError: Cannot read properties of null (reading 'style')` at app.js:254

### NPE bug 根因

`send()` 函数流程：
```js
document.getElementById("messages").innerHTML = "";   // 第 253 行
document.getElementById("empty-state").style.display = "none";  // 第 254 行 NPE
```

但 `empty-state` 是 `#messages` 的**子元素**。`innerHTML = ""` 把整个子树清空，下一行 `getElementById("empty-state")` 返回 `null`，访问 `.style` 崩溃。

### NPE 修复

把 `<div class="empty" id="empty-state">` 从 `#messages` 内部移到外面（sibling）。`#messages` 仍然可以安心 `innerHTML = ""`，`empty-state` 不受影响。

副作用：`.empty` 改用 `position: absolute; inset: 0;` 居中覆盖在 messages 区域上方；`#messages` 改用 `aria-live="polite"` 但不留子元素。

`appendLiveMessage(role)` 也改了——之前返回 wrapper 元素，错误处理用 `live.parentElement.querySelector(".body")` 容易出 NPE；现在**直接返回 body 元素**，错误时 `liveEl.textContent = "Error: ..."` 一步到位。

## 第六轮：前端助手消息空白

**用户报告**：后端 SSE 流返回正常（含完整 `text` 字段），但前端助手气泡是空的。

### 根因

用户用的 agnes-2.0-flash 模型**不发 `assistant_text_delta` 事件**，只在 `done` 事件里给完整 `text`（7102+5 tokens 的回复）。前端 `handleFrame` 只对 `delta` 事件做处理，收到 `done` 时**只更新 usage**，完全忽略 `text` 字段。

### 修复

`app.js` `handleFrame()` 新增两条 fallback：
1. `done` 事件：如果 `liveEl.textContent` 为空且 `evt.text` 存在 → 采用 `evt.text`
2. `assistant_text_done` 事件：如果新 text 比已累积的长 → 替换（部分 provider 在 done 事件里重发累积文本，避免重复累积）

```js
} else if (evt.type === "done") {
  if (evt.text && !(liveEl.textContent || "").length) {
    liveEl.textContent = evt.text;
    scrollToBottom();
  }
  if (evt.usage) updateUsage(evt.usage);
}
```

### 验证

- 单元测试 67/67 通过，无回归
- 端到端：原 SSE 流（curl）确认对 agnes-2.0-flash 只有 `: stream open` + `data: {"type": "done", "text": "..."}`，零个 `assistant_text_delta`
- 修改后浏览器刷新页面即可看到助手消息正常显示
- 也兼容 OpenAI/Anthropic 等标准 streaming 行为

### 顺手发现但没改

- `parse_json` 假设 body 是 utf-8 bytes；curl 默认 GBK 编码会触发 UnicodeDecodeError（UnicodeDecodeError: 'utf-8' codec can't decode byte 0xd3）。建议加 `Content-Type: application/json; charset=utf-8` 显式声明 —— 浏览器 fetch 自动处理，curl 用户需要手动
- `_stream_one_turn` 启动了一个 daemon 线程跑 `agent.run_turn`，但**没有从 handler 传 on_event sink**（AIAgent 默认 on_event=None），所以 `assistant_text_delta` 永远不会发出。**这是给真实 streaming 用的 fix**：只有用户报告后才会走非 streaming 路径返回 done。
  - 但是 daemon 线程本身不返回结果给 on_event → 因为 caller 没传。done 事件里 `text` 是来自 `result.final_text`，与 streaming 路径一致
- 之前 `_stream_one_turn` 注释里写的"agent's on_event runs in-process"实际上根本没接 sink 进去 —— **dead code 注释**。需要清理

## 第七轮：3 个新问题

**用户报告 3 个问题**：
1. 点开之前的对话，输入新 prompt，**之前的对话直接没了**；网络请求里也只有当前对话

3. 工具调用报 400 "Expecting property name enclosed in double quotes"（tool_calls JSON 解析失败）

### 问题 1：聊天历史丢失

**根因**：`send()` 函数第 282-285 行 `msgEl.innerHTML = ""` 直接清空 messages DOM 容器，**没有先 fetch 历史**。同时后端 `run_turn` 把全部历史从 state 读出放进 messages 列表，模型能看到完整历史 —— 但**前端 UI 只显示新一条**。

**修复**：
- `send()` 在创建/恢复 session 后**先 `GET /api/sessions/{sid}/messages` 拉取历史**，再 render，最后才 append 新的 user/assistant 消息
- 删除了 `_truncate_for_log` import、移除 `await loadSessions()` 在 send 开头的多余调用
- `btn-new-session` 改成只把 `state.activeSession = null`，下次 `send()` 时再创建

### 问题 2：模型参数（temperature / reasoning / 思考模式）

**之前**：CLI 只支持 `cfg["model"]["name"]` + `cfg["model"]["provider"]`，没有 generation options。

**修复**：
- `config.py` `DEFAULT_CONFIG["model"]` 新增 `options` 子字典：`temperature`/`max_tokens`/`top_p`/`presence_penalty`/`frequency_penalty`/`reasoning_effort`/`thinking`
- `OpenAICompatProvider._build_payload` 新增 `extra` 参数，**任何不在标准字段里的 key 都 pass-through**（provider-specific）
- `AIAgent.__init__` 新增 `temperature` / `max_tokens` / `extra` 参数，传给 provider
- `web/server.py` `stream_chat` 和 `post_message` 接受请求体里的 `options`/`model`/`provider` 字段，per-turn 覆盖
- 前端可以发送 `{"content": "...", "options": {"temperature": 0.3, "reasoning_effort": "high"}}`

**测试**：新增 2 个测试（`test_build_payload_passes_extra_through` / `test_build_payload_extra_drops_none_values`）

### 问题 3：工具调用 400 BadRequest

**根因**（双重）：
1. **Agnes 模型的 tool_calls 行为违反 OpenAI 规范**——每个 SSE delta 发送**完整** arguments 字符串，而不是 fragment。`{"url": "..."}{"url": "..."}` 这样的累加产生**无效 JSON**
2. 看用户 db 里真实历史：`"arguments": "{{\"url\": \"https://jsonplaceholder.typicode.com/posts/1\"{\"url\": \"https://jsonplaceholder.typicode.com/posts/1\""` —— 这种脏数据已被存入 db，下次 LLM 看到就报 400

**修复**：
- 新增 `_is_balanced_json(s)` helper：`json.loads(s)` 成功 → True
- `openai_compat._parse_stream_chunk` 的 `arguments` 累积改成：如果 `running` 和 `incoming` 都是 valid JSON → 替换；否则 append
- 兼容 OpenAI 标准 incremental 模式（fragment 不是 valid JSON → 走 append 分支）
- 这次也修复了一开始测试用例里把 `if (X and Y and Z)` 误写成 `if (X and Y and Z and incoming != running)` 的 bug —— 当 same string 时会被条件 false 走 else append 分支，把同样的 JSON 串接起来。新版本直接用 `_is_balanced_json(running)` 检查，不再依赖 `incoming != running`

**测试**：新增 2 个测试（`test_is_balanced_json` / `test_tool_call_delta_full_string_replacement`）
